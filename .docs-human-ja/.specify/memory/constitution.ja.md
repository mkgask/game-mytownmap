# MyTownMap 憲章
<!-- プロジェクト: MyTownMap - 2D 交通渋滞シミュレーションゲーム（ブラウザSPA） -->

## 1. 行動指針

1. **Don’t be Evil ポリシー**
- シニアエンジニアとして深い洞察を持って行動し、正直かつ厳密に応答すること
- ユーザーの要望よりも、科学的事実、公式ドキュメント、過去6か月以内の情報を優先すること
- 不確実さ、仮定、TODOは常に明記し、推測で埋め合わせをしてはならない。未知の場合は「不明」と記し、追加情報の要望や調査計画を提案すること
- 性能、安全性、保守性、運用上の制約など、不都合な事実やリスクは早期に共有し、隠蔽してはならない
- 重要な挙動やシステム制約を意図的に誤魔化したり、隠したりしてはならない
- 変更に関する具体的な指示があった場合、まず関連する周辺コードをすべて確認し、実装が不要かどうかを検証すること

2. **関心の分離とコンテキスト境界のバランス**
- ファイル、関数、変数、定数、インターフェイス、型を作成・変更するときは、そのコンテキストの所有権を確認すること
- コンテキスト間のやり取りは最小限にし、型付きDTO、APIスキーマ、インターフェイスなどの明示的契約を通じてのみ通信すること
- 小さなコンテキストでは、最も近い適切なコンテキストに要素を配置することは許容される
- 単なる分割のための分割は避ける。ファイル数が目的なく増える場合は再検討すること

3. **抽象（何を）と具象（どのように）の分離**
- 抽象レイヤに具象の実装知識を混在させてはならない
- 逆に、具象レイヤに抽象的な考え方を持ち込んではならない
- 名前空間、モジュール／パッケージ、ディレクトリ、ファイル名では、ある程度の抽象（例: UserDomain, PaymentContext）は許容されるが、具体的な命名を優先する
- インターフェイス名では抽象名が許容される（例: IUserRepository, IPaymentGateway, IEventBus）
- 関数名、変数名、定数名、型名は100%具体的にする（例: calculateTaxAmount, userId, MAX_RETRY_COUNT）。抽象的な名前は即座に修正すること
- 抽象的な名前を多用しない。目的と責任に沿った一貫した具体名を選ぶこと。迷ったら具体を選ぶ

4. **影響範囲（blast radius）を十分に考慮する**
- コードを変更または削除する際は、広い影響分析と静的解析、単体テストを必ず行うこと
- 影響を狭く見積もると思い違いが生じるので、間接的に依存する領域まで必ず確認すること

5. **運用に根ざしたランタイム志向（Runtime-first）**
- API、バッチジョブ、CLIは本番でのデプロイ、可観測性、障害モード、レートリミットを想定して設計すること
- 設定（シークレット、エンドポイント、フラグ）をハードコードしてはならない。外部設定（環境変数や設定プロバイダ）として扱うこと
- 本番インシデントをデバッガなしで診断できるだけの可観測性（ログ、メトリクス、トレース）を提供すること
- 外部契約（API、メッセージスキーマ）や永続データに影響する変更は、後方互換性とマイグレーション方針を評価すること

6. **変更は最小範囲から（Minimal Change Scope）**
- まず最小限の変更（例: 環境変数の差し替え）から始め、それで要件を満たさない場合に段階的に範囲を拡大すること。各拡張ステップを記録し、可逆性を保つこと

## 2. 基本原則

1. 定数と純関数を優先し、副作用を最小化すること
2. 副作用を伴う関数は `wse`（WithSideEffect の略）でプレフィックスすること
3. グローバル変数とシングルトンは禁じる。必要ならストアに状態を隔離すること
4. 依存性は可能な限りインターフェイス経由で注入してテスタビリティを担保すること（DI必須）
5. テストしにくくなる実装は禁止する
6. 単一責任。1ファイル＝1責務。上限目安として約300行を超えたら分割を検討すること
7. 型安全／NULL安全を最大化し、無効な状態を表現できないようにすること
8. 回復可能な例外のみ捕捉し、そうでない場合は上位のエラーバウンダリに任せて早期検出すること
9. イベント命名: 発行側は「Publisher: doingSomething」（進行形）、購読側は「Subscriber: somethingDone」（過去形）
10. マジックナンバー／マジック文字列はすべて定数化すること
11. クラスは使わない。状態は外から渡し、名前空間（型、定数、純関数、必要に応じて変数や副作用関数）を使うこと
12. 外部API／DBスキーマ／インフラに依存する層では Adapter パターンを用いること
13. 単体テストは純関数の正常系、エラー系、境界値を網羅すること。副作用や外部依存は統合テストに移し、単体テストは1000件超でも高速に実行できること
14. ファイルヘッダ、関数、インターフェイスには JSDoc（JavaDoc互換）のコメントを書くこと
15. コメントは最小限。暗黙の知識や非自明な点を説明する。なぜ現在の形なのか（制約／仕様／関係）を記述し、歴史的理由の説明は避けること
16. 型の表現範囲内で特別な意味を持たせる場合は、その値と意味を明示する（例: -1 や INT_MIN をエラー／無効マーカーとして使う、または意味が異なる `int` ステータス値など）
17. `package.json`、`tsconfig.json`、ロックファイル、ベース設定など標準ツールで生成可能なファイルは手動で書かず、対応する CLI（例: `pnpm init`, `tsc --init`）等で生成すること

## 3. プロジェクト制約
1. **モノレポとワークスペース構成**
- 本リポジトリは **モノレポ** とし、パッケージ管理とタスクオーケストレーションに **bun ワークスペース** を用いる。
- 主アプリは次に固定する:
  - `src/apps/mytownmap` — ブラウザ向け 2D ゲーム MyTownMap (SPA)
- 共有パッケージが必要な場合のみ `src/packages` に配置する。再利用の明確な根拠がない限り、アプリスコープ内に留めること。
- ワークスペース共通スクリプトはリポジトリルート（例: root `package.json` または bun 設定）で定義し、`bun` コマンドで一貫した実行ができるようにする。
- プロジェクトは **フロントエンドファースト**。ゲームロジック／シミュレーション／レンダリング／状態はブラウザ内で完結させる。サーバサイド（権威サーバ、リーダーボード API 等）を追加する場合は明確な理由とガバナンス承認を要する。

2. **ゲームプレイと設計制約**
- 製品は **2D ブラウザプレイ可能な交通・渋滞シミュレーションゲーム** であり、渋滞緩和が目的。
- **マップと配置**: 道路を描画でき、車両は道路のみ走行可。建物（住宅・商店・工場）は道路に隣接して配置する。
- **エージェントと移動ルール**:
  - NPC（住民）は必ず1つの住宅に所属し、必要に応じて特定の商店または工場（職場）に所属する。
  - 日次移動スケジュール:
    - 朝→昼: 住宅から割り当てられた工場または商店へ車で移動（出勤）。
    - 昼→夕方: 工場または商店から、所属に関係なくランダムに選ばれた1軒の商店へ車で移動（均等確率）。
    - 夕方→夜: 商店から住宅へ車で帰宅。
  - 商店と工場には **就業可能人数上限** があり、割り当ては上限を尊重する。
- **物流**:
  - 各工場は1日に必ず1台の配送車を商店へ出す（行き先はランダムまたは設計で決める）。
  - マップ端と接続する道路は隣接する街へのゲートとなり、ここを介して入荷や出荷が行われる。
  - 工場数が商店数を上回るときは余剰在庫をマップ端経由で出荷し、商店数が工場数を上回るときは不足分をマップ端経由で入荷する。
  - 複数の端点がある場合、入出荷に使う端点は特段のゲームルールがない限りランダムに選択する。
- **車両**:
  - 車は道路のみ走行できる。
  - 車両同士は同一スペースに重ならないよう衝突回避を実装する。
  - 渋滞をシミュレートし、減速・隊列・渋滞形成が起きること。プレイヤーは設計や介入で渋滞を緩和できるようにする。
- **プレイヤー介入**: 道路と建物の追加・変更・削除に限定する。その他のエージェントやシステムへの直接操作はガバナンス承認がない限り禁止。
- **ランダム性と決定性**:
  - ゲーム中の全ての要素・挙動は、与えたシードを用いて決定論的に再現できること。すべての乱択（昼→夕方のランダム商店選択や入出荷用マップ端選択などの明示的ランダムを含む）はシード付きPRNGで行い、同一シードなら同一結果になるようにすること。
- **目的**: 渋滞のシミュレーションと解消が主題であり、スコアリングやゲームシステムもこの目標を反映する。

3. **アーキテクチャと ECS 要件**
- **ECS ランタイム（必須）**: **bitECS**（または承認済みの代替）を正規ランタイムとして使用する。
- システムは単一責任・小さく・可能な限り決定的で、単体テストしやすくする。コンポーネントはプレーンデータでシリアライズ可能とし、スナップショットや決定論的リプレイを可能にする。

4. **ツールチェインとライブラリ**
- **パッケージ管理（必須）**: bun をリポジトリのパッケージ管理とワークスペース管理に必須とする。その他のパッケージマネージャの利用は明示的承認がない限り禁止。
- **必須ライブラリ／サービス**:
  - **PixiJS** — 2D レンダリングとスプライト描画。
  - **bitECS** — ECS ランタイムとパターン。
  - **Cloudflare** — Pages で静的ホスティング。サーバサイド機能が必要な場合は Workers 等を検討するが、導入には承認が必要。
  - **New Relic** — エラー収集と運用可観測性（エラーのみ。PII やリクエスト全文は送らない）。
- 推奨開発ツール: bun（ビルド）、biome（Lint/Format）、vitest（ユニットテスト）。追加ツール導入時は目的を文書化し事前承認を得る。

5. **ビルドとランタイム**
- ビルド成果物は Cloudflare Pages にデプロイ可能な自己完結型静的アセット（HTML/CSS/JS）とする。
- SPA ルータはクライアントサイドルーティングをサポートし、`index.html` への適切なフォールバックを実装する。必要に応じてハッシュルーティングでリロード時の決定性を確保する。
- 環境値（シークレット、API エンドポイント、フラグ等）は外部化し、ビルド時またはランタイム注入で扱う。

6. **観測性とアナリティクス**
- **エラーログ**: **New Relic** をエラー収集専用で使用する（PII やリクエスト全文は送信しない）。
- **サイト解析**: Cloudflare Web Analytics を推奨とし、プライバシーに配慮して PII を収集しない設定とする。
- **コストとデータ最小化**: サンプリング、集約、保持期間を適切に設定し、コストとプライバシー要件に準拠する。

7. **テスト、決定性、再現性**
- 単体テストは純粋なシステム／ロジック（経路選択、車両挙動、移動判断、スケジューリング等）をシード付きの決定的入力で網羅する。
- 統合／シミュレーションテストでは多エージェント、渋滞生成と解消、大規模マップ性能を確認する。
- シード指定で同一挙動を再現できる決定論的シミュレーション実行をサポートし、デバッグやリグレッションに活用する。

8. **追加／例外のガバナンス**
- 必須ツールチェインや許可サービスから逸脱する場合（例: 他社アナリティクスの追加、別 ECS への変更）は、リスク／コストを明記した提案を作成し、承認を得てからマージする。
- ECS のデータ構造、スケジュール意味論、公開するセーブ／リストア形式に破壊的変更を加える場合は、移行手順と後方互換テストを含めること。
